How many threads are you going to use? Specify the work that you intend each thread to perform.
- Each train will use a thread. Each train threads will monitor the loading and crossing of its train.
- The main thread will create threads for each train in the file and join train-threads as they cross and terminate.
- A queueing thread will organize each train thread into the correct station-queue at the correct queue index.
- Lastly a polling thread will poll each station-queue and designate the train-thread that should cross.
- So, overall there will be 3 + n threads where n is the number of trains.

Do the threads work independently? Or, is there an overall "controller" thread?
- The threads are not independent.
- The queueing and polling threads will need to coordinate since the queueing thread is a producer and the polling thread a consumer.
- The train threads will need to obey the polling thread so that each train departs appropriately.
- The main thread will ultimately control the proper termination and joining of the other threads.

How many mutexes are you going to use? Specify the operation that each mutex will guard.
- Mutex1 will guard that the polling thread does not poll during the queueing of a thread.
- Mutex2 will guard that only one thread uses the track at a time.
- Overall, I will use 2 mutexes in total.

How are you going to represent stations (which are collections of loaded trains ready to depart)? That is, what type of data structure will you use?
- I will use queues to represent station.
- Since I know there will be at most 100 trains, I will use an array queue.
- The array queue will be circular to facilitate efficient FIFO removal of trains.

How are you going to ensure that data structures in your program will not be modified concurrently?
- Using the queue mutex will ensure that queue thread modifications are not concurrent with polling operations.
- Using Convar2, as described below, will prevent multiple train threads from concurrently signaling completion to the main thread.

Which mutex is associated with the convar? Why?
What operation should be performed once pthread_cond_wait() has been unblocked and re-acquired the mutex?
- Convar1 will halt each train thread until the polling thread dispatches that train on the track.
    - This convar associates with the track mutex so that only one train crosses at a time.
    - After pthread_cond_wait(), the train thread will perform usleep() to mimic crossing the track.
- Convar2 will halt the main thread from joining train threads until the train crosses the track.
    - This convar associates with the track mutex so that only the one train that crossed directs the main thread to pthread_join it.
    - After pthread_cond_wait(), the main thread will perform the pthread_join on the train thread that just crossed the track.
- Overall, I will use 2 convars in total.

Briefly sketch the overall algorithm you will use. (You may use sentences such as: If train is loaded, get station mutex, put into queue, release station mutex.)
1. If train is loaded, get queue mutex, put into queue, release queue mutex.
2. Polling thread polls station-queues and will set a shared integer variable (int crosser) to indicate which train should cross.
3. Each train thread will wait to cross until 'int crosser' corresponds to its own train number.
4. After crossing, a shared integer variable (int crossed) indicates which train crossed.
5. After each train crosses, the main thread will identify which thread to pthread_join() by referencing 'int crossed'.
6. Once all trains cross, the mts program exits.
